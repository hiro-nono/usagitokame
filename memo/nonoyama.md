# 1章
- 物理環境
1. 作り方・所要時間

物理環境の環境構築では、各人のPCで設定する。
lang, db, server, lib, sdk, env, configを各人のPCで設定する。
開発段階では、それぞれでグループ間で共有した設定を行う。
保守段階であるバージョン更新・新メンバーとの共有・PC交換時での更新がある。

所要時間

- 初回構築：半日〜数日
- PC交換時：半日〜数日
- 新メンバー参加時：半日〜数日

3. 課題

開発環境はPCそのものだったため、開発者全員で環境を揃える必要があった。

- OS依存で動かない
- バージョンのタイポ
- バージョン共有の仕様書の更新遅滞による、新メンバーへの旧バージョンの共有
- PC交換時は最初から構築し直し
- ライブラリのバージョンアップ時は全員が更新

- 仮想環境
1. 作り方・所要時間

OSによる差分を解決した。=> 本番環境と同じOSを利用できる

① Linuxカーネルを共有する

③ cgroupsでCPU・メモリを制御する

カーネル：ソフトウェアとハードウェアの橋渡し

1. 作り方・所要時間
環境をコードで管理する。
コマンド１つで共有できる。

Dockerfileを書く
↓
docker build
↓

2. 解決
誰でも同じ環境を構築できる
OSだけでなく、アプリケーション層の差分を解決した。
linuxカーネルの共有のみのために高速

- コラム(k8s)

物理環境：PCに直接開発環境を構築するため、OSの違いによる問題が発生した。
VM：本番と同じOSを各開発者のPC上で動かせるようになり、OSレベルの差異を小さくした。
Docker：OSだけでなく、アプリケーションの実行環境（ランタイム、ライブラリ、設定）までコードとして定義し、チーム全員で同一の環境を再現できるようにした。

# 第1章 Docker以前

## 物理環境

### 1. 作り方・所要時間

物理環境では、開発者一人ひとりが自身のPCへ直接開発環境を構築する。

構築手順の例

1. OSをインストール（またはPCにプリインストールされたOSを利用）
2. プログラミング言語をインストール
3. データベースをインストール
4. Webサーバをインストール
5. ライブラリ・SDKをインストール
6. 環境変数や設定ファイルを編集

```
PC
└── Windows / Linux
      ├── Java
      ├── Node.js
      ├── PostgreSQL
      └── Nginx
```

所要時間

- 初回構築：半日〜数日
- PC交換時：半日〜数日
- 新メンバー参加時：半日〜数日

---

### 2. 課題

開発環境はPCそのものだったため、開発者全員で環境を揃える必要があった。

例えば

- Javaのバージョンが違う
- Node.jsのバージョンが違う
- PostgreSQLのバージョンが違う
- 環境変数が設定されていない

といった理由で動作しないことが頻繁に発生した。

また

- 新メンバーが参加するたびに環境構築が必要
- ライブラリのバージョンアップ時は全員が更新
- PC交換時は最初から構築し直し

など、保守コストも高かった。

---

## 仮想環境（Virtual Machine）

### 1. 作り方・所要時間

物理PC上にVirtualBoxやVMwareなどのハイパーバイザーを導入し、その上に仮想PCを作成する。

構築手順

1. VirtualBoxをインストール
2. 仮想マシンを作成
3. メモリ・CPU・ディスク容量を割り当て
4. UbuntuなどのISOイメージからOSをインストール
5. OS上でMySQLやNode.jsなどをインストール

```
Windows
│
└── VirtualBox
      │
      └── 仮想PC
            │
            └── Ubuntu
                  ├── Node.js
                  ├── PostgreSQL
                  └── Nginx
```

所要時間

- 仮想マシン作成：5〜30分
- OSインストール：20〜60分
- ミドルウェア構築：1〜3時間

合計：約2〜4時間

---

### 2. 解決

仮想環境では、開発環境をPCから分離できるようになった。

これにより

- 開発用LinuxをWindows上で再現できる
- 開発環境を壊してもホストOSへ影響しない
- VMファイルをコピーすることで環境を複製できる

ようになった。

---

### 3. 課題

仮想環境では、仮想PCの中でさらにOSを動かしている。

```
物理PC
    ↓
仮想PC
    ↓
Ubuntu
    ↓
アプリケーション
```

そのため

- OSを毎回インストールする必要がある
- 起動に数分かかる
- メモリを大量に消費する
- ディスク容量も大きい

という課題が残った。

# 第2章 Docker

## Docker

### 1. 作り方・所要時間

DockerではOSをインストールしない。

必要な環境をDockerfileへ記述し、イメージを作成する。

例

```dockerfile
FROM node:22

WORKDIR /app

COPY . .

RUN npm install

CMD ["npm", "start"]
```

ビルド

```bash
docker build -t sample .
```

起動

```bash
docker run sample
```

また、複数コンテナを利用する場合はDocker Composeを利用する。

```bash
docker compose up
```

所要時間

- Dockerインストール：5〜10分
- 環境構築：数秒〜数分

---

### 2. 仕組み

Dockerは仮想PCを作る技術ではない。

Linuxカーネルを共有し、アプリケーションだけを隔離して実行する技術である。

```
Windows
    │
WSL2
    │
Linux Kernel
    │
Docker Engine
    │
┌────────────┐
│ Container  │
├────────────┤
│ Node.js    │
│ PostgreSQL │
└────────────┘
```

仮想マシンとの違いは、OS全体ではなく「アプリケーション実行環境」だけを持つ点にある。

そのため

- 起動が高速
- メモリ消費が少ない
- 軽量

という特徴を持つ。

---

### 3. 解決

Dockerは

「OSを配布する」のではなく、

「実行環境そのものをイメージとして配布する」

という考え方を採用した。

これにより

- 環境構築が数秒〜数分で終わる
- 新メンバーはDockerをインストールするだけ
- 全員が同じ環境を利用できる
- CI/CDでも同じ環境を再現できる

ようになった。

---

### 4. 課題

Dockerは1台のサーバで利用する分には非常に便利である。

しかし、実運用では

- コンテナを何十・何百と起動したい
- サーバ障害時に自動復旧したい
- 負荷に応じて自動で増減したい

といった要求が生まれた。

Docker単体ではこれらを管理する仕組みは提供していない。

---

## コラム Kubernetes

Dockerが「コンテナを作る技術」なら、Kubernetesは「コンテナを管理する技術」である。

例えば3台のサーバがあるとする。

```
Server1
├── Container A
├── Container B

Server2
├── Container C

Server3
├── Container D
```

Kubernetesはこれらをまとめて管理し、

- コンテナが停止したら自動で再起動
- サーバ障害時に別サーバへ移動
- アクセス増加時はコンテナ数を自動で増やす
- 不要になれば自動で削減する

といった運用を自動化する。

Dockerが「コンテナを動かす技術」であるのに対し、Kubernetesは「大量のコンテナを安定して運用する技術」である。
